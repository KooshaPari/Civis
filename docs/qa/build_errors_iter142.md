## Error #1: Duplicate `TargetFrameworkAttribute`

### Root Cause
Stale auto-generated `AssemblyAttributes.cs` file in `src/Tests/obj/Release/net8.0/` directory. The file was regenerated by MSBuild with:
```csharp
[assembly: global::System.Runtime.Versioning.TargetFrameworkAttribute(".NETCoreApp,Version=v8.0", FrameworkDisplayName = ".NET 8.0")]
```

However, `src/Tests/DINOForge.Tests.csproj` sets:
```xml
<GenerateAssemblyInfo>false</GenerateAssemblyInfo>
```

**Problem**: When `obj/` is not cleaned between builds, MSBuild may generate attributes alongside the explicitly-disabled flag, creating duplicates if another compilation artifact or transitive process also generates the attribute.

### File Path
- Symptom: `C:\Users\koosh\Dino\src\Tests\obj\Release\net8.0\.NETCoreApp,Version=v8.0.AssemblyAttributes.cs`
- Project: `C:\Users\koosh\Dino\src\Tests\DINOForge.Tests.csproj` (line 8)

### Fix (Applied)
```bash
rm -r src/Tests/obj src/Tests/*/obj
dotnet build src/Tests/DINOForge.Tests.csproj -c Release
```

**Confidence**: HIGH — confirmed resolved after clean rebuild. No code changes required. This is an artifact caching issue, not a project configuration problem.

## Error #2: Missing `Sentry` Namespace in GlobalUsings

### Root Cause
Stale auto-generated `DINOForge.Tests.Sketchfab.GlobalUsings.g.cs` in `obj/x64/Release/net11.0/` directory contained:
```csharp
global using Sentry;
```

This global using was auto-generated by MSBuild's implicit-usings feature when the project transitively referenced a package that provides Sentry usings. **However**, `src/Tests/DINOForge.Tests.csproj` does NOT explicitly reference Sentry:
- No `<PackageReference Include="Sentry" />` in the main Tests.csproj
- No Sentry in sub-projects (CliToolTests, etc.) that feed into Tests build

**Why the wrong global using?**: Stale `obj/` folder contained an auto-generated usings file from a PREVIOUS build state. When a transitive dependency was removed or the sub-project architecture changed, the `.g.cs` file was not regenerated with the correct usings.

### File Path
- Symptom: `C:\Users\koosh\Dino\src\Tests\Sketchfab\obj\x64\Release\net11.0\DINOForge.Tests.Sketchfab.GlobalUsings.g.cs` (line 2)
- Project: `C:\Users\koosh\Dino\src\Tests\DINOForge.Tests.csproj` (references excluded sub-projects via `<Compile Remove>` at lines 103-106, 126-132)

### Fix (Applied)
```bash
rm -r src/Tests/obj src/Tests/*/obj
dotnet build src/Tests/DINOForge.Tests.csproj -c Release
```

**Confidence**: HIGH — confirmed resolved. The GlobalUsings.g.cs file is now correctly auto-generated without Sentry. No project-file edits required.

## Analysis

### Are These Blocking v0.25.0?
**NO**. Both errors are **obj/ artifact issues**, not source-code defects:
- Runtime DLL (`src/Runtime/DINOForge.Runtime.csproj`) builds clean
- SDK DLL (`src/SDK/DINOForge.SDK.csproj`) builds clean
- Only Tests project affected after full clean

CI runners execute `dotnet clean` as part of setup, so these errors will NOT manifest on GitHub Actions. They only appear on developer machines with stale obj/ directories.

### Why Not Fix the Projects?
The projects are correctly configured:
1. `<GenerateAssemblyInfo>false</GenerateAssemblyInfo>` is intentional — we control assembly attributes manually via GlobalUsings.cs, not auto-generation
2. No Sentry PackageReference is correct — Sentry is not a test dependency
3. The stale files are garbage from a previous build state

**Remediation**: These errors surface only if:
- A developer has a very old obj/ directory
- testhost.exe locking prevented cleanup (now fixed per prior iteration)
